## 1. 定位矩阵

### 1.1 接口概述

分析用户在「专业独特性 × 市场需求度」矩阵中的位置，给出定位反馈与行动建议。

| 项目 | 说明 |
|------|------|
| **接口地址** | `GET /api/v1/demo/position-matrix` |
| **Content-Type** | `application/json` |
| **认证方式** | 无需认证（Demo 阶段） |
| **数据源** | LLM 实时生成 |

### 1.2 请求参数

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `user_id` | string | 否 | — | 用户唯一标识 |
| `profile_id` | string | 否 | — | 画像 ID，与 `user_id` 二选一 |
| `industry` | string | 否 | — | 行业，如 `教育`、`美妆`、`职场` |
| `stage` | string | 否 | — | 阶段，如 `起步`、`成长`、`成熟` |
| `platform` | string | 否 | — | 主运营平台，如 `小红书`、`抖音`、`B站` |
| `goal` | string | 否 | — | 运营目标，如 `涨粉`、`转化`、`品牌` |
| `pain_point` | string | 否 | — | 当前痛点，如 `选题靠感觉`、`内容没流量` |

### 1.3 请求示例

**cURL**

```bash
curl -X GET "http://localhost:8000/api/v1/demo/position-matrix?\
industry=教育&stage=起步&platform=小红书&goal=涨粉&pain_point=选题靠感觉"
```

**Python**

```python
import requests

resp = requests.get(
    "http://localhost:8000/api/v1/demo/position-matrix",
    params={
        "industry": "教育",
        "stage": "起步",
        "platform": "小红书",
        "goal": "涨粉",
        "pain_point": "选题靠感觉",
    },
)
data = resp.json()
print(data["data"]["x"], data["data"]["y"])
```

### 1.4 响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "x": 72,
    "y": 78,
    "feedback": "你的定位比较清晰，有专业特色也有市场需求。",
    "suggestion": "接下来用案例内容放大你的独特性。"
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | integer | `0` 表示成功，非零表示异常 |
| `message` | string | 状态描述 |
| `data` | object \| null | 定位矩阵数据，异常时返回 `null` |
| `data.x` | integer | 横轴坐标 0-100，表示「专业独特性」 |
| `data.y` | integer | 纵轴坐标 0-100，表示「市场需求度」 |
| `data.feedback` | string | 定位现状的客观评价 |
| `data.suggestion` | string | 下一步行动建议 |

### 1.5 空态响应

当 LLM 未配置或生成失败时：

```json
{
  "code": 0,
  "message": "success",
  "data": null
}
```

## 2. 本周榜单

### 2.1 接口概述

返回本周热门选题榜单，采用「RPA 真实抓取 + LLM 增强」混合策略：
1. 通过 RPA 抓取抖音/B站/小红书的实时热门话题
2. LLM 基于真实话题 + 用户画像计算适配度并生成结构化榜单
3. 服务端二次排序与分页
4. RPA 失败时自动降级为纯 LLM 生成

| 项目 | 说明 |
|------|------|
| **接口地址** | `GET /api/v1/demo/weekly-rankings` |
| **Content-Type** | `application/json` |
| **认证方式** | 无需认证（Demo 阶段） |
| **数据源** | RPA 真实抓取 + LLM 增强 |

### 2.2 请求参数

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `sort_by` | string | 否 | `fit_score` | 排序方式：`fit_score`（适配度）或 `heat`（热度） |
| `limit` | integer | 否 | `10` | 返回数量，范围 1-50 |
| `offset` | integer | 否 | `0` | 分页偏移量，≥0 |
| `user_id` | string | 否 | — | 用户唯一标识 |
| `profile_id` | string | 否 | — | 画像 ID |
| `industry` | string | 否 | — | 行业 |
| `stage` | string | 否 | — | 阶段 |
| `platform` | string | 否 | — | 主运营平台 |
| `goal` | string | 否 | — | 运营目标 |
| `pain_point` | string | 否 | — | 当前痛点 |

### 2.3 请求示例

**cURL**

```bash
# 按适配度排序，取前 10 条
curl -X GET "http://localhost:8000/api/v1/demo/weekly-rankings?\
sort_by=fit_score&limit=10&offset=0&\
industry=教育&stage=起步&platform=小红书"

# 按热度排序，分页取第 11-20 条
curl -X GET "http://localhost:8000/api/v1/demo/weekly-rankings?\
sort_by=heat&limit=10&offset=10&\
industry=职场&goal=涨粉"
```

**Python**

```python
import requests

resp = requests.get(
    "http://localhost:8000/api/v1/demo/weekly-rankings",
    params={
        "sort_by": "fit_score",
        "limit": 10,
        "offset": 0,
        "industry": "教育",
        "stage": "起步",
        "platform": "小红书",
    },
)
data = resp.json()
for topic in data["data"]["list"]:
    print(f"[{topic['source']}] {topic['name']} (适配度: {topic['fit_score']})")
```

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [
      {
        "id": "topic_001",

        "angles": ["差异化定位", "价值主张清晰", "目标用户明确"],
        "title_templates": [
          "为什么[你的领域]需要重新定义个人IP？",
          "[你的名字]如何用3个月找到定位？"
        ],
        "warnings": []
      }
    ],
    "total": 10,
    "data_source": "rpa+llm"
  }

```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | integer | `0` 表示成功 |
| `message` | string | 状态描述 |
| `data` | object | 榜单数据 |
| `data.list` | array | 选题列表 |
| `data.list[].id` | string | 选题唯一标识 |
| `data.list[].name` | string | 选题名称（方向） |
| `data.list[].source` | string | 热点来源平台：`小红书` / `抖音` / `B站` |
| `data.list[].fit_score` | integer | 适配度 0-100，越高越契合用户定位 |
| `data.list[].heat` | integer | 市场热度 0-100，基于真实话题排名计算 |
| `data.list[].delta` | integer | 热度周环比变化 |
| `data.list[].risk_level` | string | 风险等级：`low` / `medium` / `high` |
| `data.list[].angles` | string[] | 推荐切入角度（3 个） |
| `data.list[].title_templates` | string[] | 推荐标题模板（2 个） |
| `data.list[].warnings` | string[] | 风险提示（空数组表示无风险） |
| `data.total` | integer | 总条数（未经分页的原始总数） |
| `data.data_source` | string | 数据来源：`rpa+llm`（有真实数据）或 `llm_only`（纯生成） |

### 2.6 数据来源说明

| `data_source` | 含义 | 触发条件 |
|---------------|------|---------|
| `rpa+llm` | RPA 抓取真实热门话题 + LLM 增强 | 至少一个平台抓取成功 |
| `llm_only` | 纯 LLM 生成（无真实数据） | 所有平台抓取失败或 LLM 未配置 |

## 3. 跨平台内容生成

### 3.1 接口概述

接收选题种子或用户描述，SSE 流式返回适合小红书、抖音、B站等多个平台的差异化内容版本。

直接编排 LLM + PlatformRegistry + MethodologyRegistry，不经过 orchestra 和独立 MCP Skill。

| 项目 | 说明 |
|------|------|
| **接口地址** | `POST /api/v1/services/cross-platform-content/stream` |
| **Content-Type** | `application/json` |
| **响应类型** | `text/event-stream` (SSE) |
| **认证方式** | 无需认证（Demo 阶段） |

### 3.2 请求参数

复用现有 `ServiceStreamRequest` 模型，业务字段通过 `context` 扩展：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `user_id` | string | 是 | 用户唯一标识，min_length=1 |
| `conversation_id` | string | 是 | 对话唯一标识，min_length=1 |
| `message` | string | 是 | 用户当前输入，如「生成职场穿搭内容」或「小红书的再软一点」 |
| `platform` | string | 否 | 可选平台上下文 |
| `context` | object | 否 | 业务扩展字段（见下表） |
| `mode` | string | 否 | 不使用 |

**`context` 扩展字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `target_platforms` | string[] | 否 | 目标平台列表，如 `["xiaohongshu", "douyin", "bilibili"]`，未提供则从 message 解析 |
| `content_type` | string | 否 | 内容类型：`图文`（默认）/`视频`/`仅文字` |
| `seed_topic` | object | 否 | 来自榜单的选题种子 |
| `seed_topic.id` | string | — | 选题 ID |
| `seed_topic.name` | string | — | 选题名称 |
| `seed_topic.angles` | string[] | — | 切入角度 |
| `seed_topic.title_templates` | string[] | — | 推荐标题模板 |
| `user_position` | object | 否 | 来自定位矩阵的用户坐标 |
| `user_position.x` | number | — | 专业独特性 |
| `user_position.y` | number | — | 市场需求度 |
| `user_position.feedback` | string | — | 定位反馈 |
| `master_content` | object | 否 | 若用户已提供核心内容则直接适配 |
| `master_content.title` | string | — | 核心标题 |
| `master_content.content` | string | — | 核心正文 |
| `industry` | string | 否 | 行业，用于方法论匹配 |
| `optimization_goal` | string | 否 | 优化目标：`engagement`/`conversion`/`reach` |

**Python 消费示例**

```python
import requests
import json

resp = requests.post(
    "http://localhost:8000/api/v1/services/cross-platform-content/stream",
    json={
        "user_id": "u123",
        "conversation_id": "c456",
        "message": "生成职场穿搭内容",
        "context": {
            "target_platforms": ["xiaohongshu", "douyin", "bilibili"],
            "content_type": "图文",
        },
    },
    stream=True,
)

for line in resp.iter_lines():
    if line.startswith(b"data: "):
        event = json.loads(line[6:])
        if event["type"] == "platform_chunk":
            pf = event["platform"]
            content = event["content"]
            print(f"\n=== {pf} ===")
            print(f"标题: {content['title']}")
            print(f"正文: {content['body'][:100]}...")
        elif event["type"] == "warning":
            print(f"⚠️  {event['platform']} 风险提示: {event['warnings']}")
        elif event["type"] == "done":
            print(f"\n✅ 完成，共 {event['total_platforms']} 个平台")
```

### 3.4 SSE 事件流说明

SSE 按顺序推送以下事件：

```
data: {"type": "start", "service": "cross-platform-content", "platforms": ["xiaohongshu", "douyin", "bilibili"]}

data: {"type": "platform_chunk", "platform": "xiaohongshu", "content": {...}}

data: {"type": "warning", "platform": "xiaohongshu", "warnings": ["命中medical类禁用词：疗效"]}

data: {"type": "platform_chunk", "platform": "douyin", "content": {...}}

data: {"type": "platform_chunk", "platform": "bilibili", "content": {...}}

data: {"type": "done", "total_platforms": 3, "full_length": 1580}
```

| 事件类型 | 触发时机 | 数据结构 |
|---------|---------|---------|
| `start` | 开始生成时 | `{"type": "start", "service": "...", "platforms": [...]}` |
| `platform_chunk` | 单个平台内容生成完毕 | `{"type": "platform_chunk", "platform": "...", "content": {...}}` |
| `warning` | 合规扫描命中敏感词时 | `{"type": "warning", "platform": "...", "warnings": [...]}` |
| `error` | 某平台生成失败时（不影响其他平台） | `{"type": "error", "platform": "...", "message": "..."}` |
| `done` | 全部完成时 | `{"type": "done", "total_platforms": N, "full_length": N}` |

### 3.5 各平台 content 字段说明

**通用字段（所有平台）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 平台适配后的标题 |
| `body` | string | 平台适配后的正文 |
| `hashtags` | string[] | 推荐标签列表 |
| `hook` | string | 视频平台的黄金 3 秒钩子（图文平台可能为空） |
| `best_time` | string | 建议发布时间 |
| `compliance_warnings` | string[] | 合规风险提示 |
| `platform` | string | 平台标识：`xiaohongshu` / `douyin` / `bilibili` |
| `format` | string | 内容形式：`图文` / `视频` |

**小红书特有字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `pic_count_tip` | string | 推荐图片数量，如 `6-9 张` |
| `pic_ratio` | string | 推荐图片比例，如 `3:4` |

**抖音特有字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `duration_tip` | string | 推荐视频时长，如 `15-60 秒` |

**B站特有字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `danmu_prompt` | string | 弹幕引导语建议 |

```json
    "danmu_prompt": "在关键转折点设置弹幕互动引导"
  }
}
```

### 3.7 多轮对话修稿说明

系统会自动检测修稿意图。当用户消息包含以下关键词时，触发修稿模式：

```
改、调整、再、修、软一点、硬一点、短一点、长一点、
换、重写、优化、润色、改一下、不够、太、稍微、能不能、可以
```

**修稿流程：**
1. 从历史记忆中找到上一轮该平台的生成结果
2. 将上一轮内容 + 新的修稿要求传给 LLM
3. LLM 在保持核心信息不变的前提下做风格/语气/结构调整
4. 重新推送 `platform_chunk` 事件

**限制：**
- 修稿仅基于最近一轮的历史内容
- 跨 session（不同 `conversation_id`）无法修稿

## 4. 通用规范

### 4.1 数据流转关系

```
┌─────────────┐     ┌─────────────┐     ┌─────────────────────┐
│  ① 定位矩阵  │ ──► │  ② 本周榜单  │ ──► │  ③ 跨平台内容生成   │
│  (我在哪)    │     │ (写什么)    │     │ (怎么写+多平台适配) │
└─────────────┘     └──────┬──────┘     └─────────────────────┘
                           │
                           │ "去改写" 按钮
                           ▼
                    透传选题数据：
                    - seed_topic (id, name, angles, title_templates)
                    - user_position (x, y, feedback)
```

### 4.2 用户画像上下文

三个接口共用同一套画像参数，前端可缓存并在接口间透传：

| 参数 | 示例值 | 用途 |
|------|--------|------|
| `industry` | `教育` / `美妆` / `职场` | 行业定位 |
| `stage` | `起步` / `成长` / `成熟` | 发展阶段 |
| `platform` | `小红书` / `抖音` / `B站` | 主运营平台 |
| `goal` | `涨粉` / `转化` / `品牌` | 运营目标 |
| `pain_point` | `选题靠感觉` / `内容没流量` | 当前痛点 |

### 4.3 降级策略

| 场景 | 降级行为 | 用户感知 |
|------|---------|---------|
| LLM Hub 未初始化 | 定位矩阵返回 `data: null`；榜单返回空列表 | 页面显示空态或提示 |
| RPA 抓取全部失败 | 榜单 `data_source: "llm_only"`，纯 LLM 生成 | 无感知（内容质量可能略降） |
| 某平台生成失败 | SSE 推送 `error` 事件，继续其他平台 | 看到该平台生成失败提示，其他平台正常 |
| LLM JSON 解析失败 | 返回 fallback 默认数据 | 无感知 |

## 5. 错误码说明

### 5.1 REST 接口错误码

| HTTP 状态码 | `code` | 含义 | 处理建议 |
|-------------|--------|------|---------|
| 200 | `0` | 成功 | — |
| 200 | `0` | 成功但 `data: null` | LLM 未配置，检查 `infra/config/llm.yaml` |
| 400 | — | 参数错误 | 检查请求参数是否符合规范 |
| 500 | — | 服务端内部错误 | 查看服务端日志 |

### 5.2 SSE 接口错误事件

SSE 流中通过 `error` 事件返回错误，不会中断 HTTP 连接：

```json
{"type": "error", "message": "LLM Hub 未初始化。请检查服务启动日志。"}
```

| 错误信息 | 原因 | 解决方式 |
|---------|------|---------|
| `LLM Hub 未初始化` | `infra/config/llm.yaml` 缺失或格式错误 | 检查配置文件是否存在且格式正确 |
| `无法获取 LLM 客户端` | `llm.yaml` 中未配置对应 skill | 使用默认客户端或补充 skill_config |
| `LLM 客户端缺少 API Key` | 环境变量未设置 API Key | 在 `.env` 中设置 `DEEPSEEK_API_KEY` 或对应 Key |
| `生成失败: ...` | 某平台生成过程中异常 | 单平台失败不影响其他平台，可重试 |
